A Model Context Protocol server for driving **Cisco Packet Tracer 9.0**
from outside. A small custom extension hooks into PT's editor and opens
a local HTTP bridge; the MCP server then calls PT's native IPC API
through it to build topologies, configure routers, run simulations and
so on. Written in TypeScript on Bun.

> [!WARNING]
> **Experimental project.** This is version 0.1.0, built in spare time
> as a personal experiment. It works against PT 9.0.0.0810 on my
> PC and passes the smoke tests, but it is not production-ready.
> Expect bugs, API changes, and things that break the next time PT
> updates. Issues with what you find are very welcome.

## Why another PT MCP?

Community projects for PT 6.x–8.x relied on JavaScript helpers shipped
by a signed third-party extension. PT 9.0 shut that door: App Meta
Files (`.pta`) now have to be ECDSA-signed by Cisco, so no open-source
project can ship one. The remaining option is to talk to PT's webview
from outside, and that is what this server does:

- A small **custom `.pts` extension** is installed from PT's Extensions
  menu. All it does is run a polling loop against the local HTTP bridge.
- When the MCP server needs to write, it calls **PT's native IPC API**
  directly: things like
  `ipc.appWindow().getActiveWorkspace().getLogicalWorkspace().addDevice(...)`
  or `getDevice(...).getCommandLine().enterCommand(...)`. No
  intermediate libraries, no opaque JS copied from elsewhere.
- Every piece of JS that ends up running inside PT's Script Engine is
  generated by this repo. If you want to know what is being sent, look
  in `src/ipc/`.

## Philosophy: canvas-first

Other tools keep a "plan" object in memory, validate it, and finally
push it to the simulator. Not here:

- **There is no `TopologyPlan` in memory.** Whatever is in PT is what
  exists; the canvas is the plan.
- **Every recipe re-reads a snapshot of the canvas before acting.** Run
  the same recipe twice on a half-built canvas and the second pass
  finishes the bits that were missing instead of duplicating devices.
  You get idempotency without having to think about it.
- **Validation queries the live canvas**, not an internal struct. When
  something is wrong (duplicate IPs, peers in different subnets, a
  powered-off router) you can see it in PT, not just in an abstract
  graph.
- **Persistence stores snapshots, not plans.** Going back means diffing
  the current canvas against an older dump and seeing what changed.
  There is no magic rehydration step.

The longer version lives in [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md).

## Requirements

- Cisco Packet Tracer **9.0** (macOS, Windows or Linux).
- [Bun](https://bun.sh) `>=1.2`. Node alone is not enough: the code is
  TypeScript and runs directly under Bun, with no separate compile step.
- The `mcp-bridge.pts` extension installed in PT. It ships with the
  repo at `extension/dist/`.

> [!TIP]
> **Why Bun and not Node?** Bun runs `.ts` files natively, starts in
> milliseconds, and bundles fetch, HTTP, and a test runner without
> extra dependencies. If you only have Node, install Bun with
> `curl -fsSL https://bun.sh/install | bash` (or `brew install bun` on
> macOS). It coexists with Node without stepping on it.

## Quick start

```bash
git clone https://github.com/jcorderop02/packet-tracer-mcp.git
cd packet-tracer-mcp
bun install
bun run start              # streamable-HTTP on :39001 (default)
# or:
bun run src/index.ts --stdio   # stdio mode for MCP clients that prefer it
```

In Packet Tracer 9:

1. `Extensions > Scripting > Configure PT Script Modules > Add` and
   pick `extension/dist/mcp-bridge.pts`.
2. Mark the extension active.
3. Open the window from `Extensions > MCP Bridge` once per PT session.

Polling kicks off on its own as soon as the window loads. To check
that everything is in place, run `pt_bridge_status`: it should reply
`connected: true` in under a second.

The full install walk-through lives in [`docs/BOOTSTRAP.md`](docs/BOOTSTRAP.md).

The MCP endpoint is:

```
http://127.0.0.1:39001/mcp
```

> [!NOTE]
> **Why `:39001`.** PT 9 already runs its own native IPC (undocumented,
> with a Cisco-signed handshake) on `:39000`. To avoid the clash and
> make it obvious the MCP server is a separate thing, this one listens
> on `:39001`. Override it with the `PACKETTRACER_MCP_PORT` environment
> variable if it gets in your way.

  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \

It must return a `result` with `serverInfo.name = "packet-tracer-mcp"`.

their `inputSchema`.

### Tests

```bash
bun test
```

The unit suite covers subnetting arithmetic, canvas inspection, the
snapshot parser, diffs, blueprint validation, and every recipe's CLI
builders. **You do not need a running PT to run the tests.**

### End-to-end walk-through

With the bridge connected (`pt_bridge_status` → `connected: true`), a
full session looks roughly like this:

```text
1. pt_list_recipes                          # see what's available
2. pt_forecast recipe='chain' params={routers:3, pcsPerLan:2}
                                            # dry-run estimate, zero changes
3. pt_cook_topology recipe='chain'
       params={routers:3, pcsPerLan:2, routing:'ospf'}
                                            # builds the lab; idempotent
4. pt_inspect_canvas                        # findings: duplicate IPs, etc.
5. pt_explain_canvas                        # human-readable narrative
6. pt_save_snapshot name='chain-baseline'   # capture T-0
7. pt_run_cli device='R1' command='show ip route'
                                            # quick CLI sanity check
8. pt_diff_snapshots before='chain-baseline'
                                            # what's changed since T-0
9. pt_save_pkt path='/abs/path/lab.pkt'     # save as native .pkt
```

If something went sideways along the way, `pt_mend_canvas` applies
conservative repairs (turning on devices that are cabled but powered
off, that kind of thing) and tells you what it would not touch.

## Capabilities

`packet-tracer-mcp` exposes **57 MCP tools** grouped by domain.

### Bridge & diagnostics

`pt_bridge_status`, `pt_ping`, `pt_clear_canvas`.

### Catalog & discovery

`pt_list_devices`, `pt_list_modules`, `pt_list_recipes`,
`pt_list_snapshots`, `pt_get_device_details`.

### Live workspace inspection (read-only)

`pt_query_topology`, `pt_inspect_canvas`, `pt_explain_canvas`,
`pt_forecast` (recipe dry-run), `pt_generate_configs` (offline IOS
CLI synthesis from a recipe blueprint — no bridge required, useful
for documentation, classroom material, or replicating the lab on
real hardware), `pt_inspect_ports` (per-port state: link/up/proto,
MAC, IP/mask, wireless flag), `pt_read_vlans` (live VLAN database
per switch), `pt_read_acl` (ACL statements via AclProcess),
`pt_show_bgp_routes` (parses `show ip bgp` into structured rows),
`pt_read_project_metadata` (description/version/filename of the
currently open .pkt).

### Canvas primitives

`pt_add_device`, `pt_add_module`, `pt_create_link`, `pt_delete_device`,
`pt_delete_link`, `pt_move_device`, `pt_auto_layout` (re-arranges the
whole canvas into a layered grid; detects transit routers and pushes
them down to a second tier), `pt_rename_device`, `pt_set_pc`,
`pt_set_device_power` (idempotent Device.setPower toggle),
`pt_add_canvas_annotation` (notes/lines/circles),
`pt_manage_clusters` (list/remove/uncluster logical clusters),
`pt_send_raw` (escape hatch: arbitrary JS in PT's Script Engine).

### Recipe-level edits (orchestration)

`pt_cook_topology`, `pt_mend_canvas`, `pt_plan_review` (mandatory
pre-flight review for ≥3-router topologies), `pt_apply_switching`,
`pt_apply_services`, `pt_configure_server_dhcp` (DHCP server on
Server-PT via native IPC API), `pt_configure_subinterface` (802.1Q
sub-interfaces for router-on-a-stick), `pt_apply_advanced_routing`,
`pt_apply_ipv6`, `pt_apply_voip`, `pt_apply_wireless`.

### Snapshots (no plans)

`pt_save_snapshot`, `pt_load_snapshot`, `pt_diff_snapshots`. Snapshots
are persisted to disk as canvas dumps. Loading one never silently
overwrites the canvas: it diffs against what you have right now and
hands the result back to you.

### Native `.pkt` persistence

`pt_save_pkt`, `pt_open_pkt` (when the server and PT share a
filesystem), plus `pt_save_pkt_to_bytes` and `pt_open_pkt_from_bytes`
for moving the `.pkt` over the network. The compressed `.pkz` format
does not work: the JS API exposes the methods but the bytes it returns
are not valid in PT 9.0.0.0810.

### Router / switch CLI

`pt_run_cli`, `pt_run_cli_bulk`, `pt_show_running` (with optional
`| section`).

### Simulation & ops

`pt_simulation_mode` (Realtime↔Simulation toggle), `pt_simulation_play`
(play/back/forward/reset), `pt_send_pdu` (originate Simple PDU),
`pt_traceroute`, `pt_screenshot` (PNG/JPG of the workspace).

## Bundled recipes

`pt_list_recipes` returns the live list. The factory ones:

| Recipe | Description |
|--------|-------------|
| `chain` | N routers in series, each with its LAN of M PCs. |
| `star` | hub router with N spokes, each spoke owning its LAN. |
| `branch_office` | HQ with one or two LANs and a branch router with its own. |
| `campus_vlan` | router-on-a-stick: one router + one switch carrying N VLANs and M PCs per VLAN. |
| `edge_nat` | edge router with PAT, DHCP, ACL, and an upstream ISP router. |
| `dual_isp` | two ISPs with eBGP + HSRP on the edge routers. |
| `voip_lab` | CME router + switch + N IP Phone 7960s; provisions telephony-service, ephone-dn, voice VLAN and DHCP option-150. |
| `ipv6_lab` | dual-stack IPv4 + IPv6 with OSPFv3. |
| `wifi_lan` | router + AccessPoint + laptops. **Limited**: PT 9's public JS API does not trigger radio association (see `docs/COVERAGE.md`). |

The first three accept `routing: 'static' | 'ospf' | 'eigrp' | 'rip' |
'none'` and `dhcp: true|false`. `campus_vlan` also accepts
`startVlanId`, `vlanStep` and `lanPool`. `wifi_lan` accepts `ssid`,
`security`, `psk` and `channel`.

## Configuration

Environment variables (all optional):

| Variable | Default | Purpose |
|----------|---------|---------|
| `PACKETTRACER_MCP_HOST` | `127.0.0.1` | MCP HTTP transport host. Use `0.0.0.0` for LAN access. |
| `PACKETTRACER_MCP_PORT` | `39001` | MCP port (avoids PT's native IPC `:39000`). |
| `PACKETTRACER_BRIDGE_PORT` | `54321` | Port the PT webview polls. |
| `PACKET_TRACER_SNAPSHOT_DIR` | `./packet-tracer-snapshots` | Snapshot persistence directory. |

## Project layout

```
src/
  bridge/        HTTP bridge polled by the PT webview
  canvas/        Snapshot, inspection, diff, subnetting arithmetic
  catalog/       PT models (54), human aliases, cable types
  ipc/           Pure string-in/string-out builders for the native IPC
                 + .pkt persistence via systemFileManager
  recipes/       Blueprints, addressing, routing, switching, services,
                 ipv6, voip, wireless, prebuilt topologies
  persistence/   Snapshot persistence
  tools/         One file per MCP tool; ALL_TOOLS aggregates them
  config.ts      Env-driven configuration
  server.ts      Lifecycle (MCP transport + bridge)
  index.ts       CLI entry point
extension/       .pts extension (PT module that runs the polling loop)
docs/            Architecture, bootstrap, coverage matrix
scripts/         smoke.ts (verification suite against real PT)
```

## FAQ

<details>
<summary><b>What is this for, exactly?</b></summary>

So an LLM (Claude, ChatGPT, Gemini — anything with an MCP client) can
create, configure and validate topologies in Packet Tracer 9 without
you touching the GUI. Useful for prepping CCNA/CCNP labs, generating
practice scenarios, or automating repetitive networking tasks.

</details>

<details>
<summary><b>Does it work without Packet Tracer running?</b></summary>

Read-only calls that don't touch the canvas (catalog, list recipes,
validate plans) work without PT. Anything that mutates the topology
(`pt_add_device`, `pt_cook_topology`, `pt_run_cli`, etc.) needs PT
open with the extension loaded.

</details>

<details>
<summary><b>Which versions of Packet Tracer are supported?</b></summary>

Only **9.0.0.0810**. The native IPC API changes between PT minor
versions and this server is calibrated against that build. Earlier
versions (8.x) don't have the modern Script Engine and won't work.

</details>

<details>
<summary><b>Why Bun instead of Node?</b></summary>

Mostly convenience. Bun runs TypeScript with no transpile step, boots
in a few milliseconds (which matters when an MCP client restarts the
server several times a day), and `bun build --compile` produces a
standalone binary you can drop on another machine without installing
anything. Node would also work; I just ended up liking the Bun flow
better for this.

</details>

<details>
<summary><b>Does it have authentication?</b></summary>

No. It assumes a trusted local environment (loopback). If you bind to
the LAN with `PACKETTRACER_MCP_HOST=0.0.0.0`, securing the port is on
you. See [`SECURITY.md`](SECURITY.md).

</details>

<details>
<summary><b>What if a recipe hangs?</b></summary>

Call `pt_clear_canvas`, lower the parameters (fewer routers/VLANs) and
try again. If you can reproduce it reliably, open an issue with the
exact parameters. More detail in
[`docs/TROUBLESHOOTING.md`](docs/TROUBLESHOOTING.md).

</details>

<details>
<summary><b>Can I add my own recipes?</b></summary>

Yes. Follow the `src/recipes/topologies/*.ts` pattern, export a
function returning a `Blueprint`, and register it in
`src/recipes/index.ts`. See [`CONTRIBUTING.md`](CONTRIBUTING.md).

</details>

### Common errors (short answers)

| Symptom | Quick fix |
|---------|-----------|
| MCP client can't connect | `lsof -i :39001` → if nothing, run `bun run src/index.ts`. |
| `pt_bridge_status` → `connected: false` | Paste the bootstrap from the README into PT's script editor and click Run. |
| Canvas not moving | Polling stopped: paste the bootstrap again. |
| `pt_run_cli` returns empty | Retry — `terminal length 0` is applied on the first `enable`. |
| `bun install` fails | `bun --version` ≥ 1.2 and `rm -rf node_modules bun.lock && bun install`. |

Full list in [`docs/TROUBLESHOOTING.md`](docs/TROUBLESHOOTING.md).

## Status

Tested against **Cisco Packet Tracer 9.0.0.0810**:

- **54 of 54** catalog models tagged `verified-pt9`.
- **57 MCP tools**, each with unit tests.
- **8 of 9 recipes** verified end-to-end (`chain`, `star`,
  `branch_office`, `campus_vlan`, `edge_nat`, `dual_isp`, `voip_lab`,
  `ipv6_lab`). The ninth, `wifi_lan`, comes up short because of a PT
  API limitation, not an MCP one.
- Layers validated end-to-end: L2 switching, L3 services (NAT, ACL,
  DHCP), dynamic routing (OSPF, BGP, HSRP, EIGRP), IPv6 with OSPFv3,
  VoIP (CME + ephone-dn), simulation and ops, and native `.pkt`
  persistence.

Detailed evidence (smoke runs, screenshots, and known limitations)
lives in [`docs/COVERAGE.md`](docs/COVERAGE.md).
